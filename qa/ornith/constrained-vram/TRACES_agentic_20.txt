**Algorithm**: The classic two-pointer approach:

- Compare elements at each pointer
- Take the smaller one and advance that pointer
- Append to result
- When one list is exhausted, append remaining elements from the other list

**Time complexity**: O(m + n) where m and n are the lengths of the two lists
**Space complexity**: O(m + n) for the result list

Let me write clean, well-documented code.

Here's a Python function that merges two sorted lists into one sorted list, using a two-pointer approach without relying on any built-in sort:

```python
def merge_sorted_lists(list1, list2):
    """
    Merge two sorted lists into a single sorted list.

    Args:
        list1 (list): First sorted list.
        list2 (list): Second sorted list.

    Returns:
        list: A new sorted list containing all elements from list1 and list2.
    """
    merged = []
    i, j = 0, 0

    # Compare elements from both lists and add the smaller one
    while i < len(list1) and j < len(list2):
        if list1[i] <= list2[j]:
            merged.append(list1[i])
            i += 1
        else:
            merged.append(list2[j])
            j += 1

    # Append any remaining elements from list1
    while i < len(list1):
        merged.append(list1[i])
        i += 1

    # Append any remaining elements from list2
    while j < len(list2):
        merged.append(list2[j])
        j += 1

    return merged
```

**Example usage:**

```python
print(merge_sorted_lists([1, 3, 5], [2, 4, 6]))   # [1, 2, 3, 4, 5, 6]
print(merge_sorted_lists([], [1, 2, 3]))           # [1, 2, 3]
print(merge_sorted_lists([1, 2, 3], []))            # [1, 2, 3]
print(merge_sorted_lists([], []))                  # []
print(merge_sorted_lists([-1, 0, 3], [-2, 1]))      # [-2, -1, 0, 1, 3]
print(merge_sorted_lists([1, 1, 2], [1, 2, 3]))    # [1, 1, 1, 2, 2, 3]
print(merge_sorted_lists([7], [3]))                # [3, 7]
```

**Key design points:**
- Uses two pointers (`i`, `j`) to track positions in each input list.
- Handles all edge cases: empty lists, unequal lengths, duplicates, and negative values.
- Maintains sorted order by always selecting the smaller current element.
- Preserves input lists (no mutation).
- Time complexity: **O(m + n)**, Space complexity: **O(m + n)** for the result.

Thread A: write counter → 6

## Diagnosis

The most likely cause is a **one-to-many relationship** between the `addresses` table and either the `customers` or `orders` table. For example, if each customer can have multiple addresses (billing, shipping, home, work), joining `orders → customers → addresses` multiplies each order by the number of addresses per customer.

### Example of the Problem

If a customer has 3 addresses, each of their 2 orders becomes 6 rows (2 × 3).

---

## Corrected Query

```sql
SELECT 
    o.order_id,
    o.order_date,
    o.total_amount,
    c.customer_id,
    c.customer_name,
    a.address_line1,
    a.address_city,
    a.address_state,
    a.address_zip
FROM orders o
JOIN customers c ON o.customer_id = c.customer_id
LEFT JOIN (
    SELECT 
        customer_id,
        address_line1,
        address_city,
        address_state,
        address_zip,
        ROW_NUMBER() OVER (
            PARTITION BY customer_id 
            ORDER BY address_id
        ) AS rn
    FROM addresses
) a ON c.customer_id = a.customer_id AND a.rn = 1;
```

### Why This Works

- The subquery selects **one address per customer** using `ROW_NUMBER()`
- The outer query joins only the selected row, eliminating duplicates

1. **Buffer overflow**: `strcpy(buf, a)` and `strcat(buf, b)` can overflow the 64-byte `buf` if `a` or `b` (or their combined length) exceeds 64 bytes.

2. **Returning address of local variable**: The function returns `buf`, which is a pointer to a local variable on the stack. When the function returns, `buf` goes out of scope and the memory is no longer valid. The caller gets a dangling pointer.

3. **Memory leak**: If the caller stores the returned pointer and later tries to free it, `free()` would be called on stack memory (undefined behavior).

The "occasionally crashes" part is key:
- The dangling pointer issue would cause a crash when the pointer is dereferenced after the function returns. This is undefined behavior - it might work sometimes (the stack frame hasn't been overwritten yet) and crash other times.
- Buffer overflow could also cause crashes, but that would be more consistent.

The most critical bug that explains "occasionally crashes" is the return of a pointer to a local variable. The stack memory gets reused, so sometimes the data is still there (works), but sometimes it's been overwritten (crashes).

The buffer overflow is also a serious bug but could be more consistent in its failure.

This function has two critical bugs:

---

**Bug 1: Returning a pointer to a local variable (the main cause of intermittent crashes)**

```c

...
return buf;  // ❌ Returns address of stack-local variable
```

`buf` is a local array on the stack. When the function returns, `buf` goes out of scope, and its memory is no longer valid. The function returns a **dangling pointer** — a pointer to memory that may be overwritten by subsequent function calls.

This explains the "occasionally crashes" behavior:
- **Sometimes it works**: The stack frame hasn't been reused yet, so the data is still there.
- **Sometimes it crashes**: The stack has been overwritten, and the caller is reading garbage memory.

**Fix**: Use `malloc()` to allocate memory on the heap:

```c
char *join(const char *a, const char *b) {
    char *buf = malloc(strlen(a) + strlen(b) + 1);
    strcpy(buf, a);
    strcat(buf, b);
    return buf;  // Caller must free(buf) when done
}
```

---

**Bug 2: Potential buffer overflow**

```c
char buf[64];
strcpy(buf, a);
strcat(buf, b);  // ❌ No bounds checking
```

If `a` or `b` (or their combined length) exceeds 64 bytes, `strcpy`/`strcat` will overflow the stack buffer. This is undefined behavior and can corrupt the stack, cause crashes, or worse — security vulnerabilities.

**Fix**: Use `snprintf()` with explicit size limits:

```c
char *join(const char *a, const char *b) {
    size_t len = strlen(a) + strlen(b) + 1;
    char *buf = malloc(len);
    if (!buf) return NULL;
    snprintf(buf, len, "%s%s", a, b);
    return buf;
}
```

# Optimistic vs. Pessimistic Locking

## Overview

Both are concurrency control strategies used to handle multiple users or processes accessing the same data simultaneously.

| Aspect               | Optimistic Locking                          | Pessimistic Locking                          |
|----------------------|---------------------------------------------|----------------------------------------------|
| **Basic Idea**       | No lock is held until commit                | Lock is acquired before any access           |
| **Concurrency**      | High — no blocking                         | Low — can block or serialize access          |
| **Conflict Handling**| Detected at commit time via version check   | Prevented at read/write time via row-level lock |
| **Performance**      | Better under low contention                 | Better under high contention                 |
| **Failure Mode**     | Retry with error                            | Deadlock risk                                |
| **Best For**         | Read-heavy workloads                        | Write-heavy or critical data                 |

## Optimistic Locking

Assumes conflicts are rare. The application reads a version number, modifies the data, and on save, checks that the version is still the same. If not, a conflict occurred and the update is either retried or rejected.

## Pessimistic Locking

Assumes conflicts are likely. The application acquires an exclusive lock on the row before reading or writing. Other transactions must wait until the lock is released.

# TypeScript Example: Optimistic Locking with a Version Column

```typescript
interface User {
  id: number;
  name: string;
  email: string;
  version: number;
}

// Simulated in-memory database
class InMemoryDB {
  private users: Map<number, User> = new Map();

  constructor() {
    this.users.set(1, { id: 1, name: 'Alice', email: 'alice@example.com', version: 0 });
    this.users.set(2, { id: 2, name: 'Bob', email: 'bob@example.com', version: 0 });
  }

  async update(userId: number, updates: Partial<User>): Promise<{ success: boolean; message: string }> {
    const user = this.users.get(userId);
    if (!user) {
      return { success: false, message: 'User not found' };
    }

    // Optimistic check: version must not have changed
    if (user.version !== 0) {
      return { success: false, message: 'Concurrent update detected — version mismatch' };
    }

    // Apply updates
    if (updates.name !== undefined) user.name = updates.name;
    if (updates.email !== undefined) user.email = updates.email;

    // Increment version
    user.version += 1;

    return { success: true, message: 'Update successful' };
  }
}

// Usage
const db = new InMemoryDB();

// ✅ First update succeeds
const result1 = await db.update(1, { name: 'Alice Updated' });
console.log(result1); // { success: true, message: 'Update successful' }

// ✅ Second update also succeeds (version was incremented)
const result2 = await db.update(1, { email: 'alice-new@example.com' });
console.log(result2); // { success: true, message: 'Update successful' }

// ❌ Simulating a concurrent update that should fail
const result3 = await db.update(1, { name: 'Should Fail' });
console.log(result3); // { success: false, message: 'Concurrent update detected — version mismatch' }
```

## Key Concepts

- **`version` column**: Tracks how many times the record has been modified.
- **Read → Modify → Check Version → Write**: The optimistic locking pattern.
- **Conflict resolution**: On mismatch, the operation is rejected (or retried with backoff in production).
